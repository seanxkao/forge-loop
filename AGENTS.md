# Forge Loop 專案規範

這是一個**獨立**的網頁增量遊戲專案（工廠優化 ＋ 做裝 ＋ 自動戰鬥），與外層 `Codex` 工作區或任何公司專案**無關**。外層 `AGENTS.md` 的規範（`Notes/` 文件夾、`work-status.md`、多 repo worktree 切換等）**一律不套用於此資料夾**。

## 工作流程

- 切換話題前，必須先確認當前工作已完成，或取得使用者明確同意擱置；除非新話題與當前議題有重大關連，否則不要在尚未收尾時跳去別的任務。
- 詢問設計問題時，一次只問一個。
- 單純的數值平衡或簡單數值更動，不需要跑完整設計流程。
- 純數值調整不需要跑 `brainstorm` 或 TDD。
- UI 微調、版面調整、agent UI 調整，以及單純數值微調，都**不需要**跑 superpower `brainstorm` 流程；只有新增功能或大改架構時才需要。
- 若數值更動屬於需求／規格層級變更，仍需同步更新 `DESIGN.md`。

## 設計文件是單一事實來源

**每次更新或新增系統需求時，都必須同步寫進 `DESIGN.md`。**

- 任何玩法系統的新增、調整、移除（戰鬥、生產、製裝、關卡、素材、數值曲線、存檔、離線收益……）都要先反映在 `DESIGN.md` 對應章節，再或同時改程式。
- `DESIGN.md` 與程式碼不一致時，以「最新確認的需求」為準，並立即把文件補正。
- 純 bug 修復、重構、樣式微調不需動文件；只要是**需求／規格層面的改動**就要記。
- 數值若仍在試調階段，於文件標明「待平衡」，不必每次微調都記，但數值「架構」（階級數、公式、掉落規則）變動要記。

## 技術與執行

- Vite + TypeScript，零執行期依賴。DOM 介面 + Canvas 2D 戰鬥，固定步長迴圈，localStorage 存檔。
- 本機 PowerShell 擋 `npm.ps1`，請用 `npm.cmd`；在子目錄執行請 `Push-Location`。
- 改動後以 `npm run build`（= `tsc && vite build`）驗證型別與打包。
- 開發中文文案使用全形標點（，。：？！）；程式碼、URL、key:value 維持半形。

## 程式結構慣例

- 遊戲邏輯放 `src/game/`（純資料 + 系統函式，不碰 DOM）；繪製放 `src/render/`；介面放 `src/ui/`。
- 邏輯與呈現解耦：戰鬥事件透過 `CombatFx` 回呼拋給上層，邏輯層不直接操作畫面。
- UI 更新採雙層：`refresh()` 在使用者操作後重建含按鈕面板；`tick()` 每幀只原地更新數值／樣式，**不可**用定時 `innerHTML` 重建含按鈕的面板（會造成 hover 閃爍與 click 遺失）。
- 存檔以**現行 schema 深層合併**舊檔（`save.ts` 的 `migrate`）：缺欄補預設、動態 record 整包沿用、型別衝突該欄回退預設、解析失敗才重置。改版**不清檔**，新增欄位通常不必動 `SAVE_VERSION`。
